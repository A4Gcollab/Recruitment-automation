### Coordination protocol

- One git repo, one `main` branch.
- Branches: `agent/<role>/v<version>-<slice>` (e.g. `agent/backend/v0.1-schema`).
- Orchestrator holds `main`, merges PRs only when demo criteria pass.
- Shared contract files: `PROGRESS.md`, `CONTRACTS.md`, `.env.example`.
- Backend publishes API contracts first; Frontend implements against them. Integrations publishes library signatures and env vars to `CONTRACTS.md`.
- No agent edits another agent's surface area without a handoff note in `PROGRESS.md`.

**Cross-cutting rules:**

1. Never run `git config --local user.name` / `user.email` — enforced by `.githooks/pre-commit`.
2. Never `--no-verify`, `--force`, `--force-with-lease` to main, or `reset --hard` against anything you did not create.
3. Never cherry-pick from another agent's branch.
4. Do not create files outside your lane — raise a blocker first.
5. Every mutating PR must update the Changelog in `PROGRESS.md`.
6. PR required to merge to `main`. No direct pushes.

## 2. Build Versions

### v0.1 — Foundation + Stage-1 Email

**Demo**: HR logs in, imports applicants from Google Sheet (ApplicantSync export), sends Stage-1 screening form email to a candidate, email arrives in candidate inbox via Gmail SMTP.

| Agent | Scope |
|---|---|
| Orchestrator | Update scaffold for v2.1 (remove Resend deps, add Nodemailer), write v0.1 work specs, define Gmail SMTP env vars, update `.env.example` + `DEPLOYMENT.md` |
| Backend | Schema: `campaigns`, `candidates`, `role_configs`, `email_queue`, `stages`, `audit_log`. NextAuth v5. `POST /api/campaigns` (create), `POST /api/campaigns/:id/import` (from Google Sheet), `GET /api/candidates`, `POST /api/emails/send` (single Stage-1). Email queue table + `GET /api/cron/process-queue` (rate-limited: 20/hr, 30-60s gaps, 9am-6pm IST). Template engine with `{CandidateFirstName}`, `{RoleName}`, `{FormLink}`, `{Deadline}`. Kill switch env flag. `lib/audit.ts`. |
| Frontend | `/login`, `/dashboard` (campaign list + create), `/campaigns/:id` (candidate table + Import button), Import modal (paste Sheet URL, column mapping preview, confirm), "Send Stage-1" action per candidate, campaign create form (role name, Google Form link, Zoom details) |
| Integrations | Nodemailer transport factory (Gmail SMTP + App Password, connection verify), `lib/sheets/client.ts` (service account auth), `lib/sheets/fetchRows.ts` (import from Google Sheet, column auto-detect) |

### v0.2 — Form Evaluation

**Demo**: Google Form response appears in response sheet, system auto-evaluates it within 5 minutes, verdict (PASS/FAIL/REVIEW) shown in dashboard. HR sees Approvals Queue for REVIEW candidates and can decide PASS/FAIL.

| Agent | Scope |
|---|---|
| Backend | Schema: add `evaluation_criteria`, `form_responses` tables. `POST /api/campaigns/:id/evaluation-config` (save criteria mapping). Evaluation engine: Tier 1 deal-breakers (any fail = FAIL), Tier 2 quality checks (word count, blank detection). `GET /api/campaigns/:id/review-queue`. `POST /api/candidates/:id/verdict` (HR decides REVIEW → PASS/FAIL). |
| Frontend | Evaluation criteria setup page (map form questions to Tier 1 or Tier 2, set acceptable answers). Candidate detail with verdict card. Approvals Queue page. |
| Integrations | Google Sheets form response poller: polls response sheet every 5 min via `GET /api/cron/poll-forms`, deduplicates by last-processed row, matches respondent to candidate by email (primary) or fuzzy name (fallback). |

### v0.3 — Reminders + Stage-2

**Demo**: At T+20h after Stage-1, reminder fires for non-responders. PASS candidates receive Stage-2 interview invite. Candidate replies "Confirmed", reply detected and logged.

| Agent | Scope |
|---|---|
| Backend | Reminder cron: `GET /api/cron/check-reminders` — candidates where `stage1_sent_at + 20h < now` AND no form submission AND no reminder yet. Queues exactly one reminder. `POST /api/campaigns/:id/send-stage2` for interview invites (Zoom link/date/time from role config). |
| Frontend | Candidate timeline view. Bulk "Send Stage-2" button for PASS candidates with confirmation modal (Gate 3). Reminder status on candidate cards. Confirmed/unconfirmed indicators. |
| Integrations | Gmail API OAuth setup. Reply detector: `GET /api/cron/poll-replies` — polls inbox every 10 min, matches replies to Stage-2 threads, detects "Confirmed" keyword, flags unstructured replies. |

### v0.4 — Campaign Tracker + Polish

**Demo**: Full end-to-end campaign. Auto-created Google Sheet per campaign (18 tracker columns from PRD §6.4), live-updated. 3 HR approval gates wired. Pipeline funnel dashboard. Message logs.

| Agent | Scope |
|---|---|
| Backend | `POST /api/campaigns/:id/create-tracker` triggers sheet creation. `GET /api/campaigns/:id/funnel` returns stage counts. Gate 1 (confirm Good Fit list before Stage-1 send), Gate 2 (confirm evaluation results before Stage-2), Gate 3 (confirm Stage-2 send with Zoom details). Audit log query endpoint. |
| Frontend | Pipeline funnel visualization. Campaign detail with tracker sheet link. Gate modals at each decision point. Audit log page. Message logs page. Settings page (Gmail config, sending window, daily cap). |
| Integrations | Auto-create Google Sheet per campaign (18 columns per PRD §6.4). Live-update tracker rows on every pipeline event. Batch writes (≤10 per API call). |

3. All new env vars are in `.env.example`.
4. `PROGRESS.md` updated: version marked complete, next version's work specs drafted.
